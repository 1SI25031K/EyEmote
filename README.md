# EyEmote



# Swift Student Challenge 開発ガイドライン

このプロジェクトでは、コードの品質維持と開発プロセスの可視化のために、以下のGit運用ルールを採用します。

---

## 1. コミットの基本方針（Atomic Commit）
「小さく作って、こまめに記録する」を徹底します。
- **1コミット = 1機能（または1修正）** に留めます。
- 大きな機能を実装した後にまとめてコミットするのではなく、途中の「動く状態」ごとに記録します。
- メリット：バグが混入したときに、どこまで戻ればいいか明確になります。

---

## 2. コミットメッセージの形式
一目で変更内容がわかるよう、以下のプレフィックス（接頭辞）を使用します。

| プレフィックス | 使用タイミング | 例 |
| :--- | :--- | :--- |
| `feat:` | 新機能の追加 | `feat: 音声認識エンジンを導入` |
| `fix:` | バグの修正 | `fix: iPad回転時のレイアウト崩れを修正` |
| `refactor:` | コードの整理（機能は変えない） | `refactor: ViewModelのロジックを整理` |
| `docs:` | ドキュメントの更新 | `docs: READMEにGitルールを追加` |
| `ui:` | 見た目の調整（HIGに基づく修正など） | `ui: ボタンのカラーをアクセントカラーに変更` |
| `chore:` | 設定変更やファイル整理 | `chore: 画像アセットの追加` |

**書き方のコツ:**
- 1行目に簡潔に内容を書く。
- 審査員が「開発の苦労や工夫」を読み取れるような言葉を選びます。

---

## 3. ブランチ運用（Branching Strategy）
個人開発でも、実験的な機能追加の際はブランチを分けます。

- **`main` ブランチ**: 常に「エラーなくビルドでき、審査員に見せられる」安定した状態。
- **`feature/機能名` ブランチ**: 新しい機能を作るときに使用。完成したら `main` にマージ（統合）します。

> [!TIP]
> 複雑なアニメーションや新しいフレームワーク（Visionなど）を試すときは、必ず `feature/` ブランチを作って作業しましょう。失敗しても `main` は無傷です。

---

## 4. Xcode特有の注意点
- **Swift Playgrounds (.swiftpm)**:
  - Xcodeで見える「Sources」や「Resources」の構造を大切にします。
  - ファイルを移動した後は、一度ビルドして「リンク切れ」がないか確認してからコミットします。
- **.gitignore**:
  - 不要なキャッシュファイルや、Mac固有の隠しファイル（.DS_Store）がリポジトリに混入しないよう設定します。

---

## 5. SSC審査員へのアピールポイント
Gitの履歴は「あなたがどのように試行錯誤してこのアプリを作り上げたか」というストーリーを伝えます。
- 「なぜこの実装にしたのか」という意思決定の跡を、コミットメッセージやREADMEに残していくことを意識しましょう。
